# fix-665040a6984f0a0d — E2T1 does not call next() after vote callbacks, breaking E2T2 live results for inline button votes

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 FQABOT

E2T1.ts:11-44 handles `callback_query:data` with `vote:opt:` prefix. It records the vote via `voteStore.upsert()` and calls `ctx.answerCallbackQuery()`, but **never calls `next()`**. Since handlers are loaded alphabetically (E2T1 before E2T2), E2T2's `callback_query:data` handler at E2T2.ts:103-112 never receives these callbacks, so `tryEditPollMessage()` is never invoked. The poll message is NOT updated with live vote totals when a user votes via the inline keyboard buttons added by E2T1.

E2T2's `poll_answer` handler (E2T2.ts:114-119) still works for native Telegram poll votes, but the inline-button voting path — the core E2T1 feature — does NOT trigger live results as required by the E2T2 spec ("Edit the poll message to reflect live vote totals when votes change").

**Fix:** E2T1 must call `await next()` after handling the vote callback (after the `answerCallbackQuery` call), so the update propagates to E2T2's middleware.
