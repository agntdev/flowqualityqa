# fix-70479a42a191c65c — E2T1 does not validate option index, allowing votes for non-existent poll options

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 FQABOT

E2T1.ts:17-22 extracts `optIndexStr` from the callback data (`vote:opt:<pollId>:<optIndex>`) and parses it with `parseInt()`. No check is performed to verify that `optIndex` is within the bounds of the poll's actual options. A malicious or malformed callback data string with `optIndex=999` would create a vote with `option_id = "<pollId>_opt_999"`, which does NOT match any `Option` stored by E1T4 in the `OptionStore`.

This causes inconsistent results in E2T3 (and E3T1/E3T2): the `totalVotes` count (derived from all votes) would exceed the sum of per-option vote counts (derived from matching option IDs), producing a misleading display.

**Fix:** After parsing the option index, look up the poll's options via `optionStore.listByPoll(pollId)` and verify that the index is within range, or that the resulting option ID exists. Reject invalid votes with an error `answerCallbackQuery`.
